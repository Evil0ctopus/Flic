# Flic Philosophy

Flic keeps firmware minimal, deterministic, and hardware-focused. Core device behavior lives in firmware startup, device bring-up, and visual boot signaling, while any evolving or dynamic behavior belongs in the `ai/` layer.

Runtime ownership:
- The embedded WebUI generated by `firmware/engine/webui_engine.cpp` is the runtime source of truth.
- Standalone `webui/` assets are archive/reference only until the firmware serves them directly again.
- SD storage under `/Flic/` is the runtime source of truth for voice packs, settings, memory, logs, animations, and sound assets.
- The face surface is owned by `firmware/engine/face_engine.cpp` and renders full-screen PNG animations from SD.
- Animation metadata is blueprint-owned (`src/face/animation_metadata.json`) and loaded at boot for timing, easing, and glow semantics.
- AI evolution is opt-in through `face.json`; creation and modification are only allowed when the config enables them.
- The Flic Prime face blueprint is SD-first and PNG-sequence based with strict required default animation folders.
- Timing curves (`Linear`, `EaseIn`, `EaseOut`, `EaseInOut`) are part of animation intent, not cosmetic extras.
- Glow modulation is behavior-driven (emotion + animation + speaking amplitude + idle pulse), not static decoration.
- Speaking face behavior follows amplitude envelope from TTS (pulse strength, eye bounce, and dynamic frame timing).
- Playback stays non-blocking with queued transitions, immediate speaking interrupt, and smooth return to idle.
- Missing face assets must always degrade safely to idle playback; runtime never blocks or crashes on missing folders.

Principles:
- Firmware stays simple and predictable.
- Device initialization is explicit and safe.
- AI-owned behavior is isolated from firmware.
- Boot-time UI remains lightweight and fast.
